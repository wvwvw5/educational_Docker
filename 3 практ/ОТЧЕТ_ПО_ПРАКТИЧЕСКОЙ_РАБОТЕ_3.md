## 1. НАСТРОЙКА И ВЫВОД КОНФИГУРАЦИИ GIT

### 1.1 Настройка пользователя Git

Первым шагом необходимо настроить Git с информацией о пользователе. Это важно для идентификации автора коммитов.

**Команды для настройки:**
```bash
git config --global user.name "Студент"
git config --global user.email "student@example.com"
```

**Объяснение:**
- `--global` - настройки применяются ко всем репозиториям на компьютере
- `user.name` - имя пользователя, которое будет отображаться в истории коммитов
- `user.email` - email адрес пользователя

### 1.2 Вывод конфигурации Git

Для просмотра всех настроек Git используется команда:

```bash
git config --list
```

**Результат выполнения:**
```
credential.helper=osxkeychain
init.defaultbranch=main
filter.lfs.process=git-lfs filter-process
filter.lfs.required=true
filter.lfs.clean=git-lfs clean -- %f
filter.lfs.smudge=git-lfs smudge -- %f
user.name=Студент
user.email=student@example.com
core.repositoryformatversion=0
core.filemode=true
core.bare=false
core.logallrefupdates=true
core.ignorecase=true
core.precomposeunicode=true
remote.origin.url=https://github.com/wvwvw5/educational_Docker.git
remote.origin.fetch=+refs/heads/*:refs/remotes/origin/*
branch.main.remote=origin
branch.main.merge=refs/heads/main
branch.main.vscode-merge-base=origin/main
branch.main.vscode-merge-base=origin/main
```

**Объяснение основных параметров:**
- `user.name` и `user.email` - настройки пользователя
- `init.defaultbranch=main` - ветка по умолчанию при создании репозитория
- `core.filemode=true` - отслеживание прав доступа к файлам
- `core.ignorecase=true` - игнорирование регистра в именах файлов

## 2. СОЗДАНИЕ ЛОКАЛЬНОГО РЕПОЗИТОРИЯ

### 2.1 Инициализация репозитория

Для создания нового Git репозитория используется команда:

```bash
git init
```

**Результат выполнения:**
```
Initialized empty Git repository in /Users/ekononykhin/Desktop/учеба/4 курс/докер/3 практ/git-practice/.git/
```

**Объяснение:**
- Команда создает скрытую папку `.git` в текущей директории
- В этой папке хранится вся информация о репозитории
- Репозиторий создается пустым, без коммитов

### 2.2 Создание файлов для демонстрации

Создаем тестовые файлы для работы с репозиторием:

```bash
echo "# Практическая работа с Git" > README.md
echo "Это файл для демонстрации работы с Git" > demo.txt
```

**Результат:**
- `README.md` - файл с описанием проекта
- `demo.txt` - демонстрационный файл

## 3. РАБОТА С ФАЙЛАМИ В РЕПОЗИТОРИИ

### 3.1 Проверка статуса репозитория

Команда `git status` показывает текущее состояние репозитория:

```bash
git status
```

**Результат выполнения:**
```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	demo.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**Объяснение:**
- `On branch main` - текущая ветка
- `No commits yet` - пока нет коммитов
- `Untracked files` - файлы, которые Git не отслеживает
- Git предлагает использовать `git add` для добавления файлов

### 3.2 Добавление файлов в индекс (staging area)

Для добавления файлов в область подготовки коммита:

```bash
git add README.md demo.txt
```

**Альтернативные варианты:**
- `git add .` - добавить все файлы в текущей директории
- `git add *.txt` - добавить все .txt файлы
- `git add -A` - добавить все изменения

### 3.3 Создание коммита

После добавления файлов в индекс создаем коммит:

```bash
git commit -m "Первоначальный коммит: добавлены README.md и demo.txt"
```

**Результат выполнения:**
```
[main (root-commit) df51a1a] Первоначальный коммит: добавлены README.md и demo.txt
 2 files changed, 2 insertions(+)
 create mode 100644 README.md
 create mode 100644 demo.txt
```

**Объяснение:**
- `df51a1a` - хеш коммита (уникальный идентификатор)
- `(root-commit)` - это первый коммит в репозитории
- `2 files changed, 2 insertions(+)` - статистика изменений
- `create mode 100644` - созданы новые файлы с правами доступа

### 3.4 Модификация файлов

Добавим содержимое в существующий файл:

```bash
echo "Дополнительная информация" >> demo.txt
git add demo.txt
git commit -m "Обновлен demo.txt"
```

**Результат выполнения:**
```
[main 23ac0bb] Обновлен demo.txt
 1 file changed, 1 insertion(+)
```

## 4. РАБОТА С УКАЗАТЕЛЯМИ HEAD

### 4.1 Понятие HEAD

HEAD - это указатель на текущий коммит в рабочей директории. Он показывает, на каком коммите мы находимся.

### 4.2 Просмотр истории коммитов

```bash
git log --oneline
```

**Результат выполнения:**
```
23ac0bb Обновлен demo.txt
df51a1a Первоначальный коммит: добавлены README.md и demo.txt
```

**Объяснение:**
- Каждая строка показывает хеш коммита и сообщение
- Коммиты отсортированы от новых к старым
- `23ac0bb` - текущий коммит (HEAD указывает на него)

### 4.3 Просмотр содержимого HEAD

```bash
cat .git/HEAD
```

**Результат выполнения:**
```
ref: refs/heads/main
```

**Объяснение:**
- HEAD указывает на ветку `main`
- Это означает, что HEAD указывает на последний коммит в ветке main

### 4.4 Переключение на предыдущий коммит (detached HEAD)

```bash
git checkout df51a1a
```

**Результат выполнения:**
```
Note: switching to 'df51a1a'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.

If you want to create a new branch to retain commits you create, you may
do so (now or later) by using -c with the switch command. Example:

  git switch -c <new-branch-name>

Or undo this operation with:

  git switch -

Turn off this advice by setting config variable advice.detachedHead to false

HEAD is now at df51a1a Первоначальный коммит: добавлены README.md и demo.txt
```

**Объяснение:**
- Мы переключились на коммит `df51a1a`
- Теперь HEAD указывает напрямую на коммит, а не на ветку
- Это состояние называется "detached HEAD"

### 4.5 Проверка содержимого файла в предыдущем коммите

```bash
cat demo.txt
```

**Результат выполнения:**
```
Это файл для демонстрации работы с Git
```

**Объяснение:**
- В коммите `df51a1a` файл содержит только первую строку
- Дополнительная информация еще не была добавлена

### 4.6 Возврат к последнему коммиту

```bash
git checkout main
```

**Результат выполнения:**
```
Previous HEAD position was df51a1a Первоначальный коммит: добавлены README.md и demo.txt
Switched to branch 'main'
```

**Объяснение:**
- Мы вернулись на ветку `main`
- HEAD снова указывает на последний коммит в ветке

### 4.7 Проверка содержимого файла в текущем коммите

```bash
cat demo.txt
```

**Результат выполнения:**
```
Это файл для демонстрации работы с Git
Дополнительная информация
```

**Объяснение:**
- Теперь файл содержит обе строки
- Мы видим разницу между коммитами

## 5. ДОПОЛНИТЕЛЬНЫЕ КОМАНДЫ GIT

### 5.1 Просмотр различий между коммитами

```bash
git diff df51a1a..23ac0bb
```

**Объяснение:**
- Показывает изменения между двумя коммитами
- `+` означает добавленные строки
- `-` означает удаленные строки

### 5.2 Просмотр информации о коммите

```bash
git show 23ac0bb
```

**Объяснение:**
- Показывает детальную информацию о коммите
- Включает автора, дату, сообщение и изменения

### 5.3 Просмотр веток

```bash
git branch
```

**Объяснение:**
- Показывает все локальные ветки
- Текущая ветка выделена звездочкой

### 5.4 Просмотр удаленных репозиториев

```bash
git remote -v
```

**Объяснение:**
- Показывает настроенные удаленные репозитории
- `-v` показывает URL для fetch и push операций

В ходе практической работы были изучены основные аспекты работы с системой контроля версий Git:
